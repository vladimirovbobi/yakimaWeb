# ADR-0007: Caddy 2 as the Reverse Proxy and TLS Terminator

- **Status:** Accepted
- **Date:** 2026-05-03
- **Supersedes:** —
- **Superseded by:** —
- **Deciders:** Yakima Real Estate Hub Engineering
- **Tags:** infrastructure, networking, security, devops

## Context

ADR-0005 split the platform into a Django API and a Next.js frontend. That split forces a new question: which process sits on port 443, terminates TLS, routes `/api/*` to the API and everything else to the frontend, and applies edge concerns (rate limiting, compression, security headers, HTTP/3)?

Constraints from CLAUDE.md and the master plan:

- **Single-Caddyfile prod-like dev.** Local `docker compose up` should mirror production routing so cookie domains, CORS origins, and CSP nonces don't drift between environments. Hot-reload of routing config without a frontend or API restart.
- **Auto-TLS in production.** Solo-developer team; manual cert renewals are not on the table. Let's Encrypt automation must be built in.
- **Edge rate limiting.** Auth endpoints (`/api/v1/auth/login/`, `/api/v1/auth/refresh/`) need stricter buckets than content reads. Rate limit lives at the edge so the API isn't burning Celery workers on attack traffic.
- **Compression for content.** Blog and forum responses benefit from gzip + zstd; the frontend's static `_next/static/*` chunks are already pre-compressed by Next.js but the dynamic SSR HTML needs edge compression.
- **Predictable behavior on Railway.** Railway's edge handles TLS for HTTP services automatically, but we want a Caddyfile-as-source-of-truth for the case where ADR-0005's Vercel-fallback path activates and the same Caddyfile must run somewhere we control.

## Decision

Use **Caddy 2.x (alpine image)** as the reverse proxy in front of `frontend:3000` and `api:8000`. Caddy owns:

1. **TLS termination.** Auto-HTTPS via Let's Encrypt in production; local self-signed for `*.localhost` in dev.
2. **Routing.** `/api/*`, `/admin/*`, `/static/*` (Django collectstatic), and `/media/*` → `api:8000`. Everything else → `frontend:3000`.
3. **Edge rate limiting.** Per-IP token buckets via the `caddy-ratelimit` module.
4. **Compression.** `gzip` + `zstd` on text responses.
5. **Security headers.** HSTS, X-Frame-Options (Caddy), X-Content-Type-Options, Referrer-Policy. CSP is generated by Next.js middleware (per-request nonce; ADR-0006) and passed through Caddy untouched.
6. **HTTP/3.** Enabled by default in Caddy 2; no extra config.

This ADR is **subordinate to ADR-0005**.

## Rationale

**Alternatives evaluated.**

| Option | Verdict | Notes |
|---|---|---|
| **nginx + certbot** | Rejected | Industry standard, mature, but verbose. ~150-line config for what Caddy does in ~40. Manual cert renewal cron. Per-route rate limit needs `limit_req_zone` plus a Lua module or nginx-plus for what `caddy-ratelimit` does natively. |
| **Traefik 3** | Rejected | Similar feature set to Caddy, label-driven config (great for k8s), but our stack is plain docker compose. Traefik's docker-label routing adds an indirection that Caddyfile doesn't. Slightly more complex YAML. Toml/YAML mix. |
| **HAProxy 2** | Rejected | Excellent L7/L4 performance, but overkill for our one-API-one-frontend topology. Manual TLS automation. Higher ops literacy required for the next contributor. |
| **Cloudflare in front of Railway directly** | Rejected for v1 | Vendor lock-in for routing logic; can't run locally; rate limit rules live in dashboard not git. Could re-add Cloudflare *in front of* Caddy as a CDN at v1.1 without changing this ADR. |
| **Railway/Vercel default ingress only** | Rejected | Doesn't compose locally; rate limit and CSP nonce flow can't be tested before deploy; no fallback when ADR-0005's secondary host activates. |
| **Caddy 2.x (chosen)** | Accepted | ~40-line Caddyfile for our routing, auto-HTTPS, rate-limit module, modern HTTP/3. Single binary, alpine image is ~50 MB. Identical behavior local and prod. |

**Why Caddy specifically.**

- Caddyfile reads like prose: `reverse_proxy api:8000` instead of nginx's `proxy_pass http://api:8000;` plus six headers and a `proxy_set_header Connection "";` to avoid Keep-Alive footguns.
- Auto-HTTPS works in dev too (`tls internal` for self-signed local certs at `*.localhost`).
- `caddy-ratelimit` (v0.x but stable) gives per-path token buckets without Lua.
- HTTP/3 is on by default; no extra OpenSSL build acrobatics.
- Single static binary; no plugin-load surprises.

**Why not own everything in Caddy (e.g., CSP).** CSP needs a per-request nonce that the rendering layer (Next.js middleware in ADR-0006) emits into inline `<script>` tags. Generating the nonce at Caddy and propagating it cleanly to Next.js is more fragile than letting Next.js own the header. Caddy passes it through and adds the simpler static headers (HSTS, X-Frame-Options).

**Cost.** Caddy adds one container (~50 MB image, ~30 MB resident memory under normal load). Caddyfile is ~40 lines. No cert renewal scripts. Negligible operational overhead.

## Consequences

### Positive
- One declarative file (`caddy/Caddyfile`) is the source of truth for ingress, rate limiting, headers, and TLS.
- Auto-HTTPS in prod; auto-self-signed in dev. Same Caddyfile in both.
- Edge rate limiting protects auth endpoints before they reach Django.
- Adding a new backend service (e.g., a future webhook receiver) is a 3-line Caddyfile diff.
- HTTP/3 + zstd + gzip out of the box.

### Negative
- Smaller community than nginx; fewer Stack Overflow answers for edge cases.
- `caddy-ratelimit` is a third-party module (still pre-1.0 versioned); requires a custom build via `xcaddy` if we want it baked into the image.
- One more container in compose (8 total per ADR-0005).

### Neutral
- TLS inspection / cert debugging uses `caddy validate` and the admin API on `:2019`; different tooling than nginx's `nginx -t` workflow.
- Logs are JSON by default; standard `logrotate` doesn't apply, but log shipping to Better Stack is unchanged.

## Implementation notes

- Image: `caddy:2-alpine` for the base case. If `caddy-ratelimit` is required (Phase 1 hardening), switch to a custom image built with `xcaddy build --with github.com/mholt/caddy-ratelimit`. Pin the digest.
- `caddy/Caddyfile` (sketch):

  ```caddyfile
  {
      email ops@yakimarealestatehub.com
      servers {
          protocols h1 h2 h3
      }
  }

  yakimarealestatehub.com, www.yakimarealestatehub.com {
      encode zstd gzip
      header {
          Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"
          X-Content-Type-Options "nosniff"
          X-Frame-Options "DENY"
          Referrer-Policy "strict-origin-when-cross-origin"
          -Server
      }

      @api path /api/* /admin/* /static/* /media/*
      reverse_proxy @api api:8000

      reverse_proxy frontend:3000

      rate_limit {
          zone auth {
              match path /api/v1/auth/*
              key {remote_host}
              events 10
              window 1m
          }
          zone read {
              match path /api/v1/posts/*
              key {remote_host}
              events 120
              window 1m
          }
      }
  }
  ```

- Dev: a `Caddyfile.dev` variant uses `tls internal` and `localhost` host names so `https://yakimarealestatehub.localhost` works without browser warnings (after trusting Caddy's local CA via `caddy trust`).
- Compose service:

  ```yaml
  caddy:
    image: caddy:2-alpine
    ports: ["80:80", "443:443", "443:443/udp"]  # /udp for HTTP/3
    volumes:
      - ./caddy/Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy_data:/data
      - caddy_config:/config
    depends_on: [frontend, api]
  ```

- Health check: `curl --fail http://localhost:80/healthz` proxies to `api:8000/healthz`.
- Reload without restart: `docker compose exec caddy caddy reload --config /etc/caddy/Caddyfile`.
- Logs: JSON to stdout, shipped to Better Stack via the platform sidecar.

## Risks and mitigations

- **Risk:** `caddy-ratelimit` module breaks on a Caddy 2.x minor upgrade. **Mitigation:** pinned image digest; module pinned to a specific tag; smoke test in CI hits the rate-limited endpoints and asserts 429 above the bucket.
- **Risk:** Caddy auto-HTTPS attempts a Let's Encrypt cert on a domain whose DNS isn't pointed yet (e.g., a staging URL). **Mitigation:** explicit `tls internal` on staging hosts; production hosts only added once DNS is live.
- **Risk:** Header-of-truth confusion (CSP from Next.js, HSTS from Caddy). **Mitigation:** ADR-0006 owns CSP; this ADR owns HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy. Documented in `docs/RUNBOOK.md` "Header ownership" section.
- **Risk:** HTTP/3 misbehavior on some corporate networks. **Mitigation:** Caddy automatically falls back to HTTP/2 then HTTP/1.1; no client-side action needed.

## Validation

The decision is validated when:
- `docker compose up -d` produces a green Caddy health check within 10 s.
- Production deploy serves a valid Let's Encrypt cert within 60 s of DNS pointing.
- Load test (k6, 200 RPS for 60 s) shows Caddy 429s above the configured bucket on `/api/v1/auth/*` and lets `/api/v1/posts/*` through.
- Lighthouse "Avoid enormous network payloads" passes (zstd/gzip working).
- HTTP/3 negotiated on Chrome/Edge from a residential connection.

We revisit if: `caddy-ratelimit` is abandoned upstream; if Cloudflare's free tier makes their managed WAF + rate-limit cheaper than the operational cost of self-managing Caddy; or if traffic volume exceeds what a single Caddy container can serve (would push us to a horizontally scaled ingress).
