## Findings

### Dead code (backend)

**DEB-001 — Unused Python imports across `apps/`** • Med • backend
- Where: 28 files in `apps/`
- Current state: 28 `F401 imported but unused` (e.g. `apps/accounts/api/views.py:36 IsVendor`,
  `apps/accounts/tasks.py:7 send_mail`, `apps/operations/views.py:4 ContentType`)
- Fix applied: Yes — `ruff check apps/ --select F401 --fix --unsafe-fixes` (28 fixed)
- Savings: ~28 LOC; cleaner imports

**DEB-002 — Legacy template views/urls orphaned by Caddy edge routing** • High • backend
- Where: `apps/{accounts,content,forum,marketplace,operations,tools,core}/{views.py,urls.py}`
- Current state: 7 legacy `views.py` (684 total LOC) + 7 legacy `urls.py` wired in `config/urls.py`
  but Caddy `/realtor/* /blog/* /community/* /services/* /ops/* /tools/*` routes go to **frontend**,
  not API. Templates dir (~30 .html files) only reachable via:
  - `apps/core/sitemaps.py` `reverse(item)` — but `/sitemap.xml` is also routed to frontend
    (which has its own `sitemap.ts`)
  - `apps/core/tests/test_smoke.py` — green only because pytest hits Django directly
  - Models' `get_absolute_url()` still call `reverse("content:post_detail", ...)`, which
    backend code uses internally (e.g. RSS feeds), so URLs must remain registered
- Fix applied: Yes — see closeout block at the end of this doc for full delta.
- Savings: 591 LOC views.py + 76 LOC urls.py + 33 templates + ~1100 LOC HTML
  (~1.8K total LOC across 39 deleted files)

**DEB-003 — Top-level `package.json` + `vite.config.js` + `static/` are Phase-1 monolith** • Med • backend
- Where: `/package.json`, `/vite.config.js`, `/static/src/`, `/static/dist/`, `/postcss.config.js`,
  `/tailwind.config.js`
- Current state: `package.json` only declares Vite + AlpineJS + HTMX + Lucide + Lenis. No npm script
  references it from CI. Frontend lives in `frontend/`. Static dist file (`main.DHuZKkqo.js`) is shipped
  by whitenoise but only loaded by legacy templates (themselves orphaned, see DEB-002)
- Fix applied: No — Tracked-as-issue (depends on DEB-002 cleanup)
- Savings: 80KB+ npm tree, 6 root config files, ~250 LOC

**DEB-004 — `apps.delivery` + `apps.notifications` missing from `[tool.setuptools].packages`** • High • backend
- Where: `pyproject.toml`
- Current state: Both apps in `INSTALLED_APPS` (`config/settings/base.py:LOCAL_APPS`) but absent from
  `[tool.setuptools] packages = [...]`. `pip install -e .` will not include their migrations or
  templates. Works in dev only because Django auto-discovers via `apps/`-on-path
- Fix applied: No — Tracked-as-issue (low blast radius today, but breaks `python -m build` artifact)
- Savings: prevents future packaging regression

### Database query hot spots

**DEB-009 — N+1 on Post.tags M2M in public list/detail** • High • backend
- Where: `apps/content/api/views.py` `PublicPostListView.get_queryset()` + `PublicPostDetailView.get_queryset()`
- Current state: `PostListSerializer` and `PostDetailSerializer` include `tags = TagSerializer(many=True, ...)`
  but querysets only `select_related("author", "author__realtor_profile")`. List of 20 posts with 3
  tags each = 1 + 20 = 21 queries (or worse with Tag.posts.count() fallback)
- Fix applied: Yes — added `.prefetch_related("tags")` to both querysets
- Savings: ~20 queries per `/api/public/v1/posts/?limit=20` request, ~3 queries per detail request

**DEB-010 — `Tag.get_post_count` falls back to live count when no annotation** • Med • backend
- Where: `apps/content/api/serializers.py:31-38 TagSerializer.get_post_count`
- Current state: `obj.posts.filter(...).count()` runs per-tag if `post_count_annotated` is absent.
  Today only `PublicTagListView` annotates; tag inline inside `PostListSerializer.tags = TagSerializer(many=True)`
  does NOT. With prefetch from DEB-009 we already short-circuit the .filter join, but the .count()
  is still per-tag
- Fix applied: No — Tracked-as-issue. Would need to drop `post_count` from the inline `tags` field
  in `PostListSerializer` (cards rarely display per-tag totals) or annotate via Subquery in the
  prefetch. Lower priority: 3 .count() per page is acceptable through Sprint 2

### Cache misses

**DEB-011 — Public REST endpoints lack `Cache-Control: public, max-age=...`** • Low • backend
- Where: `apps/content/api/views.py`, `apps/marketplace/api/views.py`, `apps/forum/api/views.py`
  public list views
- Current state: No HTTP cache headers; frontend ISR (`revalidate: 600`) handles edge caching but
  Caddy/CDN can't cache the API response. For 10K MAU + Cloudflare R2 + Caddy edge, a 60-300s public
  cache header would offload backend
- Fix applied: No — Tracked-as-issue (needs `Vary: Accept, Cookie, Origin` and per-endpoint TTL
  decisions)
